## 使用者已確認的產品方向

### 遊戲化核心概念

使用者決定用「島嶼 + 孵化生命 / 守護獸 / 怪獸」作為資產紀錄的遊戲化主題。

目前共識：

- 島嶼代表目前財務狀態。
- 孵化生命、守護獸或怪獸代表里程碑、習慣、重要事件。
- 不要只用資產規模決定等級，避免一開始輸入大量歷史資產就直接跳到最高等。
- 里程碑可以有：個人、家庭總資產、淨資產、資產多元化、配息 / 利息、持續紀錄、財務目標等。
- 負債里程碑簡化，只做紀念型或鼓勵型，例如還款過半、清償，不作為主要養成壓力來源。

### 初始建檔 / 島嶼前傳

使用者已同意以下規則：

1. 初次使用時可以輸入過去資產、負債、交易、配息等歷史資料。
2. 初始建檔期間不重複播放解鎖動畫，也不把舊資料當成每日連續紀錄。
3. 使用者手動按「完成初始建檔」後，系統才建立 baseline。
4. 完成初始建檔後，如果補輸入更早的歷史資料，不自動重播全部動畫，只提示「島史新增片段」，可讓使用者手動觀看。
5. 更正舊資料不應刪除已獲得成就，除非未來特別設計重算機制。

### 帳戶與名稱注意事項

目前 app 內部仍有 `me`、`wife`、`family` 等概念，但未來使用者名稱可以自訂，例如可以叫「多拉A夢」。

未來規劃：

- 免費版：一個個人帳戶，可自由命名 + 一個總資產視圖。
- 其他個人帳戶可能作為付費解鎖。
- 程式設計上不要把顯示名稱當成固定 ID。
- 遊戲資料應盡量使用穩定 ID / scope，不要硬綁「我 / 太太」文字。

### 遊戲化底層規格

核心方向定版為「資產生命誌 / Asset Chronicle」。這不是單純的成就徽章系統，而是「自然生態 + 航誌 + 孵化」的財務生命紀錄。第一版遊戲化入口定為「島嶼蛋」：完成初始建檔後取得一顆蛋，後續財務行為產生不同養分，養分組合決定孵化方向。孵化生命、守護獸或怪獸不是一開始固定，而是由養分與島嶼區域自然長出來。

命名與世界觀原則：

- 對外與核心概念避免使用「財富島、共富島、資產島、理財島、金錢島」這類直白名稱。
- 目前定版主軸使用「資產生命誌 / Asset Chronicle」。
- 內部仍可保留島嶼、棲地、島史、航誌、年輪、蛋、養分、守護獸、怪獸、未知生命等元素。
- 歷史人文可以作為文案風格、主題包或華語版彩蛋，但不綁死底層規則。
- 多語言策略後續再細化；目前先避免使用過度依賴單一文化典故的底層名稱。

重要平衡原則：

- 不獎勵高頻操作本身，避免讓每天記帳或頻繁交易的人天然比較強。
- 不懲罰低頻使用者，月度整理、季度回顧、年度盤點也應能推進。
- 不鼓勵為了解鎖而交易；買賣可以進島史，但交易次數不是主要成長來源。
- 長期持有、低交易頻率、穩定盤點，也要有完整成長路線。
- 初始建檔只是島嶼前傳，不把大量舊資料一次轉成大量解鎖。
- 負債只做正向紀念與管理鼓勵，不做扣分、羞辱或壓力型怪物。
- 無配息、無負債、資產種類較少的使用者，也不能被卡住主線。

島嶼成長採多軸線，而不是單一路線：

| 軸線 | 島嶼區域 | 代表意義 | 適合的使用者行為 |
|---|---|---|---|
| 盤點軸 | 起始海灘 | 看清楚財務起點與島史 | 初始建檔、補登、整理舊事件 |
| 紀錄軸 | 記帳森林 | 穩定回來整理，不要求每天 | 月度整理、季度回顧、重要事件記錄 |
| 成長軸 | 成長山丘 | 相對 baseline 的淨值改善 | 長期持有、儲蓄、資產成長 |
| 配置軸 | 配置港口 | 資產結構逐漸清楚 | 分類、配置、再平衡，不強迫多買 |
| 現金流軸 | 收成果園 | 配息、利息、租金等收入流 | 記錄現金流；非必修主線 |
| 減壓軸 | 減壓溫泉 | 負債被看見、管理與下降 | 還款進度、單筆清償、總負債歸零 |

島嶼蛋與養分系統：

- 完成初始建檔後，使用者取得第一顆 `first_island_egg`。
- 初始建檔資料只形成島嶼前傳，不直接把蛋孵出來。
- 完成建檔後的新紀錄、整理、成長與回顧會先形成「可收穫養分」；使用者點選收穫後，養分才進入蛋。
- 蛋的狀態建議分為：`sleeping`、`glowing`、`cracking`、`hatched`。
- 不要求使用者取得所有養分；累積多種養分即可推進，避免只偏向某一類財務行為。
- 第一版可先用「累積任意 3 種養分讓蛋裂開、累積一定養分點數後孵化」作為方向，實際門檻後續再微調。

成就牆 / 可收穫養分：

- 養分不應只靠背景自動計算後直接進蛋，尤其成長養分容易受行情、匯率、補登資料影響而亂跳。
- 正式方向改為：財務資料變化 → 系統判斷達成條件 → 出現在成就牆 / 可收穫清單 → 使用者手動點選收穫 → 養分進入蛋。
- 這能讓使用者理解「為什麼得到這份養分」，也方便未來在收穫時播放動畫。
- 成就牆分為簡單與複雜兩層：
  - 簡單成就：完成初始建檔、收進第一段島史、完成第一個新月份整理、確認淨值相對基準改善、新增第一筆現金流、把負債納入管理。
  - 複雜成就：連續或累積季度整理、配置穩定一段時間、淨值改善維持多次整理、負債下降到階段門檻、島史橫跨多年、現金流累積達階段門檻。
- 記錄、回顧、成長養分都應改成「可收穫」而非直接加點；避免畫面或點數在使用者未理解時自己跳動。
- 測試期間可以保留「重置島嶼測試資料」功能，只重置島嶼、蛋、養分、島史與成就牆，不刪除資產、負債、配息或交易資料。

初始蛋類型：

- 完成初始建檔時，第一顆蛋可依「起始島地」與資產類型分布產生不同傾向，不是所有人都拿到完全一樣的蛋。
- 初始蛋類型只由完成初始建檔當下的起始資料決定，不代表好壞，也不是等級高低。
- 舊資產規模或類型只影響蛋的外觀 / 氣息 / 起始傾向，不直接孵化，也不直接發大量養分。
- 舊資料少或剛開始的人也會取得蛋，可偏「初芽 / 未定型」方向。
- 後續收穫的養分會影響孵化方向，不應被初始蛋類型完全鎖死。
- 第二顆、第三顆蛋先保留為未來擴充，不納入第一版規則。
- 可能類型：
  - 現金比例高：安定蛋。
  - 股票 / 基金比例高：林丘蛋。
  - 不動產 / 黃金比例高：地脈蛋。
  - 有配息 / 利息歷史：果實蛋。
  - 有負債且納入管理：溫泉蛋。
  - 資產類型均衡：潮汐蛋或混合蛋。
  - 資料很少、剛開始：初芽蛋。

養分類型：

| 養分 | 來源 | 平衡方式 | 可能孵化傾向 |
|---|---|---|---|
| 記錄養分 | 完成建檔後的新月份整理、重要事件紀錄 | 一個月份最多推進一次，不做每日打卡壓力 | 森林系生命、記錄型特徵 |
| 回顧養分 | 查看並收進島史片段、整理補登事件 | 多筆補登可進島史，但不靠大量補登刷進度 | 島史系生命、回憶型特徵 |
| 成長養分 | 淨值相對 baseline 改善，進成就牆後由使用者收穫 | 以百分比為主、金額為輔；不看短期行情跳動，需經整理或確認 | 山丘系守護獸幼體 |
| 配置養分 | 資產分類清楚、配置結構建立、使用再平衡 | 不鼓勵為了解鎖亂買，只看結構是否更清楚 | 港口系領航員、航海型角色 |
| 收成養分 | 配息、利息、租金等現金流紀錄 | 非必修主線，沒有現金流策略也不會卡住 | 果園系生命、收成型特徵 |
| 減壓養分 | 負債被納入管理、下降、清償 | 只做正向鼓勵，不做壓力或扣分 | 溫泉系生命、安定型守護獸 |

孵化方向：

- 主要養分不直接等於固定角色，而是決定孵化傾向。
- 記錄養分較高：偏森林系，可能是書記、記錄員、觀察者。
- 回顧養分較高：偏島史系，可能是史官、收藏者、回憶守護者。
- 成長養分較高：偏山丘系，可能是守護獸幼體或拓地型角色。
- 配置養分較高：偏港口系，可能是領航員、航線管理者。
- 收成養分較高：偏果園系，可能是照料員、採收者。
- 減壓養分較高：偏溫泉系，可能是管家、安定型守護獸。
- 若養分均衡，角色可走「混合型島嶼生命」或「混合型守護獸幼體」方向，避免系統硬把使用者貼單一標籤。

升級、轉換與提示：

- 孵化後的角色可以升級：同一條養分線持續深化，例如書記型角色逐步成長為森林書記、年輪守望者。
- 孵化後也可以轉換或長出特徵：若後續其他養分變強，角色可轉職或出現混合特性。
- 未孵化與未進化階段不直接列完整條件，可用剪影、傳聞、聲音、氣味或短提示維持期待感。
- 範例提示：
  - 「蛋殼裡傳來紙頁翻動的聲音。」
  - 「蛋殼邊緣有微微草葉香。」
  - 「牠似乎在等待下一次月度整理。」
  - 「港口的風還不夠穩。」
  - 「果園還缺第一場收成。」

孵化生命、守護獸與怪獸的分工：

- 孵化生命：從島嶼蛋誕生的小型生命型態，代表日常習慣、區域狀態、重要事件。名稱、物種、美術都先保留彈性，不預設成人形社群角色。
- 守護獸：大型區域里程碑，代表某條軸線進入成熟階段。
- 怪獸：保留給未來更大型、較有戲劇性的財務目標或長期挑戰，不急著放進第一版。
- 島史：時間線與回憶錄，負責保存「這座島如何走到現在」。
- 圖鑑不是第一個入口。第一版應先呈現島嶼蛋與養分狀態，生命型態 / 守護獸圖鑑在孵化後自然展開。

每個區域先設 3 級進度，不一次塞太多名稱：

| 區域 | Lv1 | Lv2 | Lv3 |
|---|---|---|---|
| 起始海灘 | 完成初始建檔 | 查看並收進第一段島史片段 | 島史橫跨 2 個以上年份 |
| 記帳森林 | 完成建檔後第一個新月份有整理 | 連續或累積 3 個月份有整理 | 一年內 4 個季度都有整理 |
| 成長山丘 | 淨值相對 baseline 小幅成長 | 淨值相對 baseline 穩定成長 | 解鎖成長山丘守護獸 |
| 配置港口 | 有清楚分類的資產 | 有 2 種以上資產大類或使用再平衡 | 配置狀態穩定並解鎖港口守護獸 |
| 收成果園 | 第一筆配息 / 利息 / 租金 | 累積多筆現金流事件 | 累積現金流達階段門檻 |
| 減壓溫泉 | 有負債資料且被納入管理 | 任一筆負債下降到某階段 | 單筆清償或家庭總負債歸零 |

防偏重與防濫用規則：

- 同一天多筆事件可以進島史，但紀錄軸最多算一次整理。
- 同一月份多次新增，不應讓月度 / 季度習慣線暴衝。
- 交易次數不直接給主要進度；完整記錄成本、賣出結果、配置狀態才有意義。
- 配息 / 利息是收成果園路線，不是整個島嶼主線必修。
- 資產多元化只看結構逐漸清楚，不鼓勵為了解鎖而買不需要的資產。
- 淨值成長用相對 baseline，不用絕對資產規模比較使用者；門檻應以百分比為主、金額為輔，避免小資產使用者壓力過大，也避免大資產使用者因日常波動過快觸發。
- 成長養分應透過成就牆由使用者手動收穫，不在背景自動進蛋。
- 更正舊資料不應刪除既有孵化結果；未來若做重算，也要先設計溫和的回溯規則。

首頁與遊戲區分工：

- 首頁永遠以財務儀表板為主，優先呈現目前查看對象、總資產、淨值、損益、趨勢、配置與現金流等資訊。
- 首頁只保留小型島嶼蛋狀態提示與「查看島嶼」入口，不把完整遊戲體驗塞進首頁。
- 未來新增獨立「島嶼」頁，放完整遊戲體驗：地圖、蛋、養分、成就牆、區域、生命、守護獸。
- 島史負責時間線與回顧，可以先用 modal，未來視內容量獨立成頁。
- 圖鑑等孵化後再自然出現，不作為第一入口。

第一階段實作策略：

1. 先完成底層資料模板：`ISLAND_ZONES`、`EGG_STATE`、`NUTRIENT_TYPES`、孵化方向、區域等級、組合事件。
2. 第一版入口先做島嶼蛋與養分狀態，不先做完整圖鑑。
3. 下一版應把目前自動養分改為「成就牆 / 可收穫養分」：先挑 2 到 3 條最穩的軸線接入，包含起始海灘、記帳森林、成長山丘。
4. 配置港口、收成果園、減壓溫泉先保留資料模板，等前面流程順了再接。
5. 完成初始建檔時可先決定第一顆蛋的初始類型 / 氣息，但不直接孵化、不大量發養分。
6. 首頁只保留小型狀態提示與入口；遊戲內容逐步搬到獨立島嶼頁。
7. 孵化後再展開生命型態 / 守護獸圖鑑，且名字、美術、物種都可後改；底層穩定 ID 與養分規則先穩。
8. 美術、角色圖、動畫、島嶼場景晚一點再做，避免規則還沒穩就重畫。

## Asset Chronicle 後續完善工作清單

### 總原則

開發順序應為：先做財務資料安全與 baseline，再做成就判斷，再做養分收穫，最後才做島嶼視覺與孵化。不要一開始就做完整遊戲頁，否則會讓首頁與核心財務判斷失焦。

Asset Chronicle 不是遊戲主系統，而是 Family Assets 的低干擾敘事層。所有功能必須避免鼓勵高頻交易、每日焦慮打卡、資產規模比較、負債羞辱與資料刷成就。首頁仍以財務儀表板為主，島嶼系統只能作為溫和入口與回顧層。

### 第一階段：核心資料基礎

1. 建立 baseline 機制

- 目標：讓系統知道哪些資料屬於初始建檔前、哪些資料屬於完成建檔後，以及後續成就從哪一天開始計算。
- 需保存概念：`baseline_completed_at`、`baseline_net_worth`、`baseline_asset_snapshot`、`baseline_liability_snapshot`、`baseline_currency`、`baseline_version`。
- 實作可沿用目前 `gameState.baseline` 巢狀結構，不一定要拆成 top-level 欄位。
- 使用者按下「完成初始建檔」前，不發放大量養分。
- 完成後補登早期資料，只進入「島史新增片段」。

2. 所有遊戲物件使用穩定 ID

- 目標：避免日後改名稱導致資料壞掉。
- 需要穩定 ID 的物件：`account_id`、`asset_id`、`liability_id`、`transaction_id`、`dividend_id`、`achievement_id`、`nutrient_id`、`egg_id`、`island_event_id`、`chronicle_entry_id`。
- 顯示名稱可以改，但 ID 不可變。

### 第二階段：初始島嶼蛋

3. 建立第一顆蛋資料模型

- 完成初始建檔後，產生第一顆島嶼蛋。
- 蛋資料包含：`egg_id`、`egg_type`、`egg_state`、`created_at`、`source_baseline_version`、`record_nutrient_points`、`review_nutrient_points`、`growth_nutrient_points`、`total_nutrient_points`、`hatch_direction_score`。
- 初始狀態路線：`sleeping` → `glowing` → `cracking` → `hatched`。
- 第一版可以只做到 `sleeping`、`glowing`、`cracking`，不急著孵化。

4. 初始蛋類型判斷

- 目標：根據 baseline 當下資料，決定蛋的起點氣息。
- 第一版先做 7 種：初芽蛋、安潮蛋、林丘蛋、地脈蛋、果息蛋、溫泉蛋、潮汐蛋。
- 文案必須強調：這不是等級，也不是好壞，只是島嶼記住了你的起點。

### 第三階段：成就牆系統

5. 建立成就資料表

- 目標：讓成就可以被偵測、待確認、收穫、封存。
- 狀態：`locked`、`eligible`、`pending_confirmation`、`harvested`、`archived`、`ignored`。
- 成就資料包含：`achievement_id`、`achievement_type`、`achievement_name`、`nutrient_type`、`condition_key`、`condition_payload`、`status`、`eligible_at`、`harvested_at`、`nutrient_points`、`source_event_ids`、`baseline_version`、`user_confirmed`。

6. 第一版成就範圍

- 記錄養分：第一份島務日誌、新月份的潮聲、三次回航、一季巡島、四季有跡、年輪守候。
- 回顧養分：第一頁島史、舊海風來信、前傳標記、回憶裝訂者、跨年地層、島嶼前傳。
- 成長養分：山丘亮了一點、第一塊新土、地基變厚、月光確認、穩定山脊。
- 暫緩：守護獸、怪獸、第二顆蛋、排行榜、每日任務、連續登入。
- 實作第一版可先每類 2 到 3 個，完整清單保留為後續擴充。

### 第四階段：成長養分安全規則

7. 成長判斷規則

- 目標：避免股價、匯率、補登資料造成成就亂跳。
- 成長養分只能根據 baseline 後資料、月結淨值、確認淨值、至少 14 天穩定或季度資料。
- 不要根據單日漲跌、即時股價、匯率短期跳動、補登舊資料、使用者改名稱觸發。
- 14 天穩定與季度資料可先寫入規格，若目前資料結構不足，第一版先用使用者確認降低亂跳風險。

8. 成長成就待確認

- 以下一律進入 `pending_confirmation`：第一塊新土、地基變厚、穩定山脊、守望高地、負淨值改善、從負淨值轉正。
- 確認文案方向：「系統偵測到你的淨值相較初始基準出現改善。這可能來自投入、儲蓄、資產價格、匯率或資料修正。若你認為這反映目前財務狀態的真實變化，可以將它收進島嶼。若只是資料更正，也可以先標記為資料修正。」
- 按鈕：確認並收穫、稍後再看、標記為資料修正。

### 第五階段：養分收穫流程

9. 可收穫養分清單

- 流程：財務資料變化 → 系統偵測成就 → 成就牆出現 → 使用者手動收穫 → 養分進入蛋 → 蛋狀態更新 → 島史新增片段。
- 不要背景自動餵蛋。
- 一定要讓使用者知道：得到什麼、為什麼得到、收穫後影響什麼。

10. 養分點數平衡

- 簡單成就：1 點。
- 進階成就：2 點。
- 複雜成就：3 到 5 點。
- 蛋狀態門檻可先設：0 到 4 為 `sleeping`，5 到 11 為 `glowing`，12 到 24 為 `cracking`，25 以上為 `hatched`；孵化第二版再開。

### 第六階段：首頁與島嶼頁 UX

11. 首頁小卡

- 目標：首頁仍然是財務儀表板，不是遊戲首頁。
- 小卡只顯示：資產生命誌、蛋狀態、可收穫養分數、最近島史片段、查看島嶼按鈕。
- 不顯示：任務列表、倒數、排行榜、連續登入、紅色警告。

12. 島嶼頁第一版

- Layout：島嶼狀態區、島嶼蛋區、養分狀態、可收穫養分、島史片段、區域入口。
- 第一版開放：起始海灘、記帳森林、成長山丘、島史。
- 暫時鎖住：配置港口、收成果園、減壓溫泉、守護獸、怪獸、圖鑑。

### 第七階段：島史系統

13. 島史資料模型

- 目標：讓重要事件被記錄，而不是只做動畫。
- 島史內容：`chronicle_entry_id`、`event_type`、`title`、`description`、`related_achievement_id`、`related_financial_event_ids`、`created_at`、`event_date`、`is_from_initial_history`、`is_user_confirmed`。
- 島史類型：初始建檔、補登片段、成就收穫、成長確認、負債管理、年度盤點。

14. 島史與成就牆銜接

- 成就被收穫後，自動生成島史片段。
- 範例：「你完成了三次回航。島嶼開始出現穩定巡看的痕跡。」
- 補登資料生成：「島史新增片段。一段較早的財務紀錄被收進島嶼前傳。」

### 第八階段：文案系統

15. 建立文案字典

- 目標：避免之後文案風格混亂。
- 分類：成就名稱、成就說明、收穫按鈕、待確認文案、空狀態文案、負債文案、成長文案、錯誤狀態文案。
- 禁用語：變有錢、財富自由、落後、輸了、失敗、負債警告、高資產玩家、低資產玩家、每日任務。

### 第九階段：防濫用與資料修正

16. 防刷規則

- 月份成就每月最多一次。
- 補登成就以月份 / 年度去重。
- 不以交易筆數給養分。
- 不以資產數量給養分。
- 不因頻繁登入給養分。
- 不因頻繁交易給養分。

17. 資料更正規則

- 原則：已收穫成就不輕易刪除。
- 如果使用者更正資料導致條件消失：已收穫成就保留；島史標記為「依當時資料形成」；不回收養分；但不再重複觸發同一成就。
- 文案：「這段島史依當時資料形成。後續資料更正不會抹去它，只會讓島嶼記住新的版本。」

### 第十階段：測試清單

18. 必測情境

- 初始建檔：使用者沒有歷史資料、使用者有大量歷史資料、完成 baseline 後補登舊資料、修改 baseline 前資料、刪除 baseline 前資料。
- 成就：每月只觸發一次、補登不會刷爆、低頻使用者仍能推進、無配息者不卡關、無負債者不卡關、資產種類少者不卡關。
- 成長：baseline 為 0、baseline 小於 0、淨值短期暴漲、淨值短期回落、匯率造成變動、股價造成變動、補登造成淨值重算。
- 文案：負債者不被羞辱、小資產者不被評分、高資產者不被過度吹捧、無交易者不被懲罰。

### 建議給 Codex 的工作順序

- Sprint 1：資料模型。建立 baseline schema、egg schema、achievement schema、nutrient schema、chronicle schema，確認穩定 ID。
- Sprint 2：baseline 與初始蛋。完成初始建檔流程、按下完成後建立 baseline、根據 baseline 決定 `egg_type`、建立 `first_island_egg`、顯示初始蛋文案。
- Sprint 3：成就偵測。實作記錄養分成就、回顧養分成就、成長養分初版、成就狀態流轉。
- Sprint 4：成就牆 UI。可收穫清單、待確認清單、已收穫紀錄、收穫按鈕、收穫後更新蛋養分。
- Sprint 5：首頁小卡與島嶼頁。首頁 Asset Chronicle 小卡、查看島嶼入口、島嶼頁第一版 layout、蛋狀態顯示、養分槽顯示。
- Sprint 6：島史。成就收穫後新增島史、補登資料新增島史片段、島史 modal、島史列表、島史詳情。
- Sprint 7：防錯與測試。防刷規則、資料更正規則、baseline 重算限制、測試案例、邊界情境。
